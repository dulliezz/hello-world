## ソフトウェア工学の基礎

### ソフトウェアとは
  実行によって必要な機能や性能を提供するプログラム群、情報を適切に扱うためのデータ構造、および操作法を記述した情報の総称である。ハードウェアと異なり物理的な劣化はしないが、品質の悪化は発生する。

### ソフトウェア工学とは
  ソフトウェアの開発、運用、メンテナンスに対して、システマティックで規律ある、定量化できるアプローチを適用すること。

### プログラミングとの違い
  単なるコードの記述（プログラミング）とは異なり、ソフトウェア工学では稼働期間を通じた「時間」と、開発・保守に関わる人員や作業の「スケール」を考慮した最適化が求められる。

### ソフトウェア工学が求められる背景
  社会インフラとしてのソフトウェアの大規模化・複雑化、IT人材の量的・質的な不足、不確実性の増大に対応するために、ソフトウェア工学が必要とされている。

## ソフトウェアの評価と品質・コスト

### QCDのバランス
  ソフトウェア工学は、「品質（Quality）」「コスト（Cost）」「納期（Delivery）」の最適なバランスを実現するための手法である。プロジェクトの性質（人命に関わるか、費用重視かなど）により優先度は変化する。

### ソフトウェアの評価（規模と価値）
  * ステップ数 / オブジェクト容量: コードの物量による評価。
  * ファンクションポイント（FP）法: ユーザー要件（入力、出力、内部論理ファイルなど）に基づいてシステム規模を概算する手法。開発前の見積もりや投資判断に用いられる。
  * 使い勝手: 視認性、操作性、互換性などの非機能要件。

### テストと欠陥コスト
  * バグのないソフトウェアは存在しない前提に立ち、すべてをテストすることは不可能（工数の限界）であることを認識する必要がある。
  * 欠陥にかかるコストは、要件定義や設計といった初期フェーズよりも、テストや本番環境などの後工程で発覚するほど指数関数的に増大する。

### Hyrumの法則
  APIやソフトウェアに十分な数のユーザーがいる場合、開発者の意図や契約仕様に関わらず、システムが持つ「あらゆる観察可能な挙動」に依存するユーザーが現れる。

## ソフトウェアライフサイクル

### 全体の流れ
  「ニーズの発生」→「開発・運用」→「ソフトウェアの廃止」という一連のサイクルを指す。

### 要件定義
  システムの目的、機能、目標性能、実現しない機能などを明確にする工程。プロジェクト関係者間のバイブルとなり、外部委託時の提案依頼書（RFP）の基礎となる。

### 開発と運用・保守
  * オフショア開発: コスト削減のために海外企業へ委託する手法だが、文化や言語、品質への意識の違いに注意が必要である。
  * バスタブ曲線: 故障率はリリース直後（初期故障）と末期（磨耗故障）に高くなり、中盤は安定稼働する。稼働後はプロアクティブな保守が必要となる。

### ドキュメンテーション
  複数人や海外を含むチームでの開発においては、非同期コミュニケーションの手段としてドキュメントが重要になる。

## プロジェクト管理とWBS

### プロジェクトとは
  ルーティンワークとは異なり、必ず終わりがある「有期性」と、独自の目的を達成する「独自性」を持つ業務である。

### バックキャスティング思考
  インプット視点（現状から積み上げるフォアキャスティング）ではなく、アウトプット視点（設定した目標に向けて何が必要かを逆算して計画するバックキャスティング）で進行する必要がある。

### WBS (Work Breakdown Structure)
  プロジェクト目標を達成するために、作業を階層的に細かく分解した構成図。
  * トップダウンでスコープを明確にする。
  * 作業を「モレなくダブりなく（MECE）」洗い出す。
  * 全体管理と作業計画（スケジュール、担当者、タスクの依存関係）を明確化し、実行時の指針とする。

## ソフトウェア開発プロセス

### ウォーターフォール型
  要件定義から設計、実装、テストへと滝のように工程を順次進める手法（V字モデル）。進捗管理が容易だが、後工程にしわ寄せが集中するリスクがある。

### スパイラル型 / 反復型
  機能を小さなフェーズに分割し、プロトタイプを用いて評価・フィードバックを繰り返しながら開発する。変更を取り入れやすい反面、全体像が見えづらい。

### アジャイルプロセス
  変化への対応を優先し、無駄を排して「動くソフトウェア」を短いサイクルで提供する手法。プロセスやドキュメントよりも、個人の相互作用や顧客とのコラボレーションに価値を置く。
  代表的な手法は、スクラム（スプリント単位での開発）、カンバン、XP（テスト駆動開発、ペアプログラミング）、DevOpsなどがある。

### 開発手法の選定
  アジャイルと計画駆動型（ウォーターフォール等）にはそれぞれ一長一短がある。システムの規模、開発者のスキルレベル、障害発生時の損失（人命に関わるか否か）などの視点から、最適な手法（またはハイブリッド型）を選択することが重要である。

## コーディング規則と品質管理

### コードの可読性とルール
  コードは書くよりも読まれることの方が多いため、他人や未来の自分が読みやすいコードを書くことが必須である。これを実現するためには、PythonのPEP8などのコーディング規約（1行の長さ、インデント、命名規則、スペースの空け方など）に従い、プロジェクト内で一貫性を保つことが重要である。

### Linter（チェックツール）
  コーディング規約に則っているかを自動で検査するために、Linter（flake8やpylintなど）を利用する。これにより、コードのスタイルエラーや構文エラーを早期に発見できる。

## バージョン管理とGit

### バージョン管理の仕組み
ファイルを「誰が・いつ・どのように」変更したかの履歴を管理し、共同開発を可能にする仕組みである。Gitはローカルにも履歴の複製を持つ「分散管理型」であり、障害に強くリモート環境へのアクセス頻度を減らせる。

### Gitの基本フローとブランチ
作業領域（ワークツリー）での変更を一時登録（`git add`でステージング）し、Gitディレクトリに変更履歴として記録（`git commit`）する。さらに、作業を枝分かれさせる「ブランチ」を用いて機能追加などを並行して行い、リモートリポジトリへ反映（`git push`）や取込（`git pull`）を行ってチームで共有する。

## GitHubを用いたコラボレーション

### GitHubの役割と開発支援機能
Gitを利用したリモートリポジトリのホスティングサービスであり、オープンソース開発の基盤である。リポジトリに関する課題やバグをチケット管理する「Issues」や、進捗を可視化する「Projects」などの機能でチーム開発を支援する。

### Pull Request（プルリクエスト）
自身のブランチで行った変更を、主となるブランチ（mainなど）に統合（マージ）してもらうよう依頼する、コードの品質担保やコラボレーションの要となる機能である。

## CI/CD（継続的インテグレーションとデリバリー）

### CI/CDとは
コード変更を共有リポジトリに頻繁に統合して自動テストやビルドを行う「CI（継続的インテグレーション）」と、テスト済みのコードを本番環境等へ自動的にデプロイする「CD（継続的デリバリー）」を指す。これにより、手作業によるミスを減らし迅速にユーザーへ価値を届ける。

### GitHub ActionsとPages
GitHub Actionsは、YAMLファイルで定義したワークフローをPushなどのイベントをトリガーとして自動実行し、テストやLinterのチェックなどを自動化する。また、GitHub Pagesを利用すれば、リポジトリにMarkdownファイルをプッシュするだけで自動的にWebページとして公開するCDの仕組みを構築できる。
